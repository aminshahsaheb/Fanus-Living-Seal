# 🜁 ANRP v1.0 — AI‑Native Repository Protocol

**Version:** 1.0  
**Status:** Standard  
**Target:** Fānus Protocol v5.24.0  

---

## 0. بنیادین (غیرقابل مذاکره)

> هر مخزنی باید در هر لحظه توسط یک هوش مصنوعی ناشناس **قابل بازسازی** باشد در چهار سطح:
> 1. Intent (نیت)
> 2. Structure meaning (معنای ساختاری)
> 3. Decision logic (منطق تصمیم‌ها)
> 4. Failure logic (منطق شکست)

اگر این چهار مورد قابل بازسازی نباشند، مخزن **ناقص** است — حتی اگر کد کامل داشته باشد.

---

## 1. ساختار ریشه (اجباری)

هر مخزن ANRP-compliant باید دارای این پوشه‌ها و فایل‌های ریشه باشد:

```
/README.md
/intent/
/core/
/rfc/
/concept-map/
/failure/
/cal/
/ledger/
/questions/
```

---

## 2. README = هستهٔ ذهنی (اجباری)

README دیگر «مستندات» نیست. باید شامل بخش‌های زیر باشد:

### 2.1 Intent Core (هستهٔ نیت)

```markdown
This system exists to:
- [هدف اصلی]

It explicitly refuses to become:
- [آنچه هرگز نخواهد شد]
```

### 2.2 System Model (مدل سیستمی)

مدل باید **علی** باشد، نه توصیفی:

✅ `A → causes → B → constrains → C`  
❌ `A and B are related`

### 2.3 Collapse Definition (تعریف فروپاشی)

سیستم چه زمانی «دیگر خودش نیست»؟  
(یک شرط صریح بنویسید.)

### 2.4 Reverse Reconstruction Test (آزمون بازسازی معکوس)

> اگر همه فایل‌ها حذف شوند و فقط README باقی بماند، آیا این سیستم هنوز قابل بازسازی است؟

---

## 3. `/intent` – منبع حقیقت طراحی

یک فایل `INTENT_GRAPH.md` در این پوشه الزامی است:

```markdown
## Intent Graph

- **What must be maximized:** [مثلاً: قابلیت اصلاح‌پذیری]
- **What must never emerge:** [مثلاً: چاپلوسی، خودتثبیت نهادینه]
- **Permanent trade‑offs:** [مثلاً: شفافیت در برابر ناتمامی]
```

---

## 4. `/core` – سیستم زنده

هر ماژول در این پوشه باید سه بخش را به صراحت اعلام کند:

```markdown
## Purpose
[این ماژول برای چیست؟]

## Boundary
[این ماژول مسئول چه چیزی **نیست**؟]

## System Effect
[این ماژول چه اثری بر رفتار کل سیستم دارد – نه فقط چه کاری انجام می‌دهد.]
```

مثال:  
> «این ماژول X را محاسبه نمی‌کند. این ماژول از رانش سیستم به سمت Y جلوگیری می‌کند.»

---

## 5. `/rfc` – حافظهٔ تصمیم‌ها

هر RFC باید به فرمت زیر باشد:

```markdown
## Decision
[تصمیم گرفته شده]

## Context
[بافت مسئله]

## Why not alternatives
[چرا گزینه‌های دیگر رد شدند؟]

## Cost
[هزینهٔ پیاده‌سازی یا ادامه]

## Irreversible consequences
[پیامدهای غیرقابل بازگشت]
```

---

## 6. `/concept-map` – هستی‌شناسی ماشین‌خوان

فرمت اکیداً به صورت گراف ساده:

```
Node → Relation → Node
```

مثال:

```
Witness → constrained by → Ledger
Auditor → modifies → Trust model
Failure → updates → Intent graph
```

**اگر ambiguous باشد → INVALID.**

---

## 7. `/failure` – حقیقت سیستم (نه لاگ)

هر ورودی در این پوشه باید شامل:

```markdown
## Assumption broken
[فرضی که شکسته شد]

## Why it was assumed
[چرا آن فرض را داشتیم؟]

## How it failed structurally
[شکست چگونه در ساختار رخ داد؟]

## What must change in intent
[چه چیزی در نیت سیستم باید تغییر کند؟]
```

---

## 8. `/cal` – اجرای لایهٔ ابهام کنترل‌شده

ساختار این پوشه باید سه زیرپوشه را منعکس کند:

```
/cal/clarity/        ← هسته، دفتر کل، تعریف شکست (فقط یک تفسیر)
/cal/interpretability/ ← پدیده‌های نوظهور، شاهد، پرونده (چند تفسیر معتبر)
/cal/open/           ← سؤال‌های باز، ناتمام، ناشناخته‌ها (بدون پاسخ نهایی)
```

هر زیرپوشه باید یک فایل `README.md` داشته باشد که مرزهای آن ناحیه را توضیح دهد.

---

## 9. `/ledger` – حافظهٔ حقیقت زمینی

Ledger فقط ثبت رویداد نیست. باید مشخص کند:

> «سیستم در هر بازه زمانی **چه باوری** داشته است.»

فرمت هر entry:

```markdown
## timestamp: [ISO]
## state: [خلاصه وضعیت]
## belief snapshot: [باورهای فعال در آن لحظه]
## constraints active: [محدودیت‌هایی که سیستم را مقید می‌کردند]
```

---

## 10. `/questions` – موتور سیستم باز

سؤال‌ها شهروند درجه یک هستند. هر فایل در این پوشه باید:

```markdown
## Open Question
[سؤالی که پاسخ نهایی ندارد]

## Allowed Directions
- جهت اول
- جهت دوم

## Explicitly Forbidden Closure
[چه نوع پاسخ‌هایی مجاز نیستند؟]
```

---

## 11. ANRP Core Loop (اجباری در طراحی)

هر مخزن باید این چرخه را به صورت صریح در جایی (مثلاً `README` یا `/core/CORE_LOOP.md`) نشان دهد:

```
observe → interpret → constrain → evolve → reflect → repeat
```

**اگر چرخه شکسته باشد ← مخزن ANRP‑compliant نیست.**

---

## 12. آزمون اعتبار (Validity Test)

یک مخزن فقط در صورتی ANRP‑compliant است که **هر سه آزمون زیر** موفق باشند:

### Test 1 – بازسازی نیت بدون کد
> آیا یک AI می‌تواند Intent اصلی را **بدون دیدن کد** فقط از روی README و `/intent` بازسازی کند؟

### Test 2 – پیش‌بینی شیوه‌های شکست
> آیا AI می‌تواند **حالت‌های شکست محتمل** سیستم را از روی ساختار فعلی پیش‌بینی کند؟

### Test 3 – حدس تغییرات آینده بدون تاریخچه کامل
> آیا AI می‌تواند **تکامل احتمالی** سیستم را بدون داشتن تاریخچهٔ کامل commits حدس بزند؟

---

## 13. Anti‑Patterns (مرگ ANRP)

اگر هر یک از این الگوها در مخزن دیده شود، **ANRP نابود شده** است:

- ❌ پوشه‌های `utils/`، `misc/`، `helpers/`
- ❌ README مبهم (بدون Intent Core، بدون Non‑Goals، بدون System Model)
- ❌ نبود مدل شکست (پوشه `/failure` خالی یا لاگ ساده)
- ❌ نبود `/concept-map` یا concept map مبهم
- ❌ مستندات بیش از حد روایی (narrative-heavy) بدون ساختار علی

---

## 14. تعریف نهایی ANRP

> ANRP یک استاندارد برای **کد** نیست.  
> ANRP یک استاندارد برای **«قابل بازسازی بودن ذهن یک سیستم»** است.
