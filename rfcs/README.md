# RFC System – Architectural Governance Layer

هر RFC یک واحد تغییر معماری است. بدون RFC تصویب‌شده، هیچ تغییری در لایه‌های زیر مجاز نیست:

- Epistemic Engine (سیگنال‌های خام)
- Policy Layer (ترجمه به مفاهیم فانوس)
- State Machine (وضعیت‌ها و گذارها)
- LEDGER System (ساختار داده و امضا)

## چرخه حیات یک RFC

| مرحله | مسئول | شرط عبور |
| --- | --- | --- |
| **پیش‌نویس (Draft)** | هر contributor | شرح مسئله و راه‌حل پیشنهادی (حداقل ۳۰۰ کلمه) |
| **نقد (Critique)** | حداقل ۲ منتقد مستقل (یا «منتقد مقیم») | نقد در `LEDGER_OF_CRITICS.md` ثبت شود |
| **بازنویسی (Revision)** | نویسندهٔ RFC | پاسخ به تک تک نقدها در بخش «تغییرات پس از نقد» |
| **تصویب (Approval)** | هستهٔ حاکمیت (فعلاً: Amin + Āyāneh + منتقد دعوت‌شده) | امضای صریح در فایل RFC |
| **پیاده‌سازی (Implementation)** | هر توسعه‌دهنده | کد در مخزن، همراه با ارجاع به RFC |
| **بازبینی پس از پیاده‌سازی** | هستهٔ حاکمیت | حداکثر ۳۰ روز پس از merge |

## استاندارد نام‌گذاری
`RFC-XXXX` که XXXX عدد چهاررقمی ترتیبی است (از ۰۰۰۱ شروع می‌شود).

## قالب RFC
هر RFC باید شامل این بخش‌ها باشد:

1. عنوان و شماره
2. چکیده (Abstract)
3. مسئله (Problem Statement)
4. راه‌حل پیشنهادی (Proposed Solution)
5. تغییرات مورد نیاز در لایه‌ها (Changes in Layers)
6. نقد این RFC (Critique of This RFC) – توسط نویسنده یا منتقد
7. تاریخ و امضا
